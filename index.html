📡 TCPDump — Packet Capture Reference COMPLETE REFERENCE ▶
Reference — Source: TCPDump Cheat Sheet
Capture · Filter · Display · Output — Full Command Reference
TCPDump is the standard CLI packet analyzer on Linux/Unix. Requires root or CAP_NET_RAW. Use which tcpdump to locate. Write to .pcap files to open in Wireshark.
CAPTURE FLAGS
tcpdump -i any # Capture on ALL interfaces (needs root) tcpdump -i eth0 # Capture on specific interface tcpdump -i eth0 -c 100 # Stop after 100 packets tcpdump -i eth0 tcp # TCP packets only tcpdump -i eth0 udp # UDP packets only tcpdump -i eth0 icmp # ICMP only (ping traffic) tcpdump -i eth0 arp # ARP packets only tcpdump -I eth0 # Monitor mode (Wi-Fi sniffing) tcpdump -r capture.pcap # Read & analyze saved capture file
FILTER EXPRESSIONS
# Host filters tcpdump host 192.168.1.1 # Any traffic to/from IP tcpdump src host 10.0.0.5 # Source IP only tcpdump dst host 10.0.0.5 # Destination IP only tcpdump net 192.168.0.0/24 # Entire subnet # Port filters tcpdump port 443 # Any traffic on port 443 tcpdump src port 80 # Source port 80 tcpdump portrange 1000-2000 # Port range tcpdump tcp dst port 22 # TCP to port 22 (SSH) # MAC/Layer 2 filters tcpdump ether host 00:11:22:33:44:55 tcpdump ether broadcast # Ethernet broadcast frames # Logical operators tcpdump src 10.0.0.1 and port 443 # AND tcpdump dst 10.0.0.1 or src port 22 # OR tcpdump port 80 and not host 192.168.1.5 # NOT tcpdump dst host 10.0.0.1 and greater 128 # Packet > 128 bytes
DISPLAY & OUTPUT FLAGS
tcpdump -n # Don't resolve IPs to hostnames (faster) tcpdump -nn # Also don't resolve port names tcpdump -v # Verbose output (TTL, checksum, etc.) tcpdump -vv # More verbose tcpdump -A # Print payload in ASCII (HTTP inspection) tcpdump -X # Print payload in hex + ASCII tcpdump -XX # Include link-layer header in hex tcpdump -e # Print link-layer header (MAC addresses) tcpdump -q # Quick/quiet — shorter output lines tcpdump -t # Suppress timestamp tcpdump -tttt # Full date+time timestamp # Write/read pcap files tcpdump -i eth0 -w capture.pcap # Write to file (for Wireshark) tcpdump -i eth0 -U -w capture.pcap # -U: write packets immediately (no buffer) tcpdump -r capture.pcap port 443 # Read file with filter
PRACTICAL EXAMPLES
# Capture all HTTP traffic (show payload in ASCII) tcpdump -i eth0 -A port 80 # Capture SSH attempts to this host (no DNS resolution) tcpdump -nn -i eth0 tcp dst port 22 # Capture all traffic EXCEPT SSH (avoid capturing your own session) tcpdump -i eth0 not port 22 # Save 30 large packets to file, no DNS resolution tcpdump -i eth0 -n greater 32 -w out.pcap -c 30 # Pipe remote traffic to local Wireshark in real time ssh user@router "tcpdump -U -s 65535 -w - 'not port 22'" | wireshark -k -i - # Find all HTTP GET requests (show source IP + URL) tcpdump -i eth0 -A 'tcp port 80 and (tcp[((tcp[12:1] & 0xf0) >> 2):4] = 0x47455420)'

🎟️ Kerberos Authentication Flow 4-STEP PROTOCOL ▶
Protocol — Source: CEH + CISSP Cheat Sheets
Kerberos — Ticket-Based Authentication (port 88)
Kerberos uses symmetric + asymmetric encryption. Default in Active Directory environments. Attacks: Kerberoasting (offline crack service ticket), Pass-the-Ticket (reuse TGT/TGS), Golden Ticket (forged TGT using KRBTGT hash).
KDC
Key Distribution Centre — central trust authority (runs AS + TGS)
AS
Authentication Service — validates identity; issues TGT
TGS
Ticket Granting Service — issues service tickets upon valid TGT
TGT
Ticket Granting Ticket — proof of identity; valid for session
1
Client → KDC (AS)
Client sends AS-REQ with username in cleartext. No password sent yet.
2
KDC (AS) → Client
AS responds with TGT encrypted with KRBTGT hash. Client decrypts AS-REP using their password-derived key. If incorrect password → cannot decrypt → fail.
3
Client → KDC (TGS)
Client presents TGT to TGS, requests service ticket for specific service (TGS-REQ).
4
KDC (TGS) → Client → Service
TGS issues service ticket encrypted with service account's hash. Client presents ticket to target service. Access granted without re-entering password.

🔬 Digital Forensics Process 6-PHASE PROCESS ▶
Process — Source: CISSP + Net+ Cheat Sheets
Identification → Preservation → Collection → Examination → Analysis → Presentation
Digital forensics produces court-admissible evidence. Evidence must be relevant, complete, sufficient, and reliable. Chain of custody documents every person who handled evidence, when, and what they did.
1
Identification
Identify potential evidence sources: devices, logs, network captures, cloud data. First responder secures and documents the scene before anything is touched.
2
Preservation
Protect evidence from alteration (write blockers). Follow Order of Volatility: CPU registers/cache → RAM → swap → disk → remote logs → paper. Issue Legal Hold.
3
Collection
Create forensic image (bit-for-bit copy) using tools like dd, FTK Imager, EnCase. Hash the image (MD5/SHA-256) immediately for integrity verification. Never work on originals.
4
Examination
Extract relevant data from the forensic image: files, metadata, deleted files (unallocated space), slack space, registry hives, logs, browser history, memory dumps.
5
Analysis
Interpret extracted data to reconstruct events: timeline analysis, correlation with other evidence, identification of IOCs, attribution of actions to users/systems.
6
Presentation / Court
Produce forensics report documenting methodology, findings, and conclusions. Expert witness testimony must be clear to non-technical judges/jury. E-discovery for legal proceedings.

💾 RAID Levels Reference STORAGE REDUNDANCY ▶
Reference — Source: A+ Cheat Sheet
Redundant Array of Independent Disks — Levels 0, 1, 5, 6, 10
RAID provides fault tolerance and/or performance. Does NOT replace backups. Minimum disk counts and failure tolerance differ by level.
LevelNameMin DisksCan LoseCharacteristicUse Case
RAID 0 Striping 2 0 Data split across disks; combined read/write speed. No redundancy. Scratch disks, video editing (speed > safety)
RAID 1 Mirroring 2 1 Exact copy on each disk. Half usable capacity. Fast reads, slower writes. OS drives, critical single-point protection
RAID 5 Striping + Parity 3 1 Data + parity distributed. (N-1) usable capacity. Rebuild from parity on failure. File servers, NAS; good balance of speed/redundancy
RAID 6 Double Parity 4 2 Like RAID 5 but two parity blocks. (N-2) usable capacity. Slower writes. Large arrays where double failure is a real risk
RAID 10 Stripe of Mirrors 4 1 per mirror Mirrored pairs then striped. 50% usable capacity. Excellent performance + redundancy. Databases, high-performance high-availability workloads
⚠️ RAID is NOT backup. RAID protects against drive failure only — not against ransomware, accidental deletion, corruption, or site disasters. Follow the 3-2-1 rule: 3 copies · 2 different media types · 1 offsite.

▸ LASER PRINTER IMAGING PROCESS (7 STEPS)
1 — Processing
Page data received; printer RIP (Raster Image Processor) converts to bitmap image in memory
2 — Charging
Primary corona wire/charge roller applies uniform negative charge (~-600V) to photosensitive drum
3 — Exposing
Laser beam neutralizes charge on drum where text/images will appear, creating a latent electrostatic image
4 — Developing
Negatively charged toner is attracted to the uncharged (exposed) areas of the drum; visible image forms
5 — Transferring
Transfer corona belt/roller applies positive charge to paper, attracting toner from drum to paper
6 — Fusing
Fuser assembly (heating element + pressure roller) melts toner permanently into paper fibers (~180°C)
7 — Cleaning
Cleaning blade scrapes residual toner off drum; erase lamp neutralizes drum charge for next cycle
