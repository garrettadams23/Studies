🔐 CIA Triad CORE MODEL ▶
Foundational Model
Confidentiality · Integrity · Availability
Every security control maps to one or more CIA pillars. The attacker's mirror is the DAD Triad: Disclosure · Alteration · Destruction/Denial.
🔒
Confidentiality
Data accessible only to authorized parties. Controls: encryption, MFA, access controls, data classification
✅
Integrity
Data is accurate & unaltered. Controls: hashing, digital signatures, checksums, version control, audit logs
⚡
Availability
Systems accessible when needed. Controls: redundancy, backups, DDoS mitigation, uptime SLAs, failover

⚠️ Risk Management Lifecycle 5-STEP CYCLE ▶
Lifecycle
Identify → Analyze → Prioritize → Treat → Monitor
A continuous cycle. Risk = Likelihood × Impact. ALE = ARO × SLE (Annualized Loss Expectancy = Annual Rate of Occurrence × Single Loss Expectancy).
1
Identify Risks
Asset inventory, threat modeling, vulnerability identification, Business Impact Analysis (BIA)
2
Analyze Risks
Qualitative (probability/impact ratings) vs Quantitative (ALE = ARO × SLE). Inherent vs residual risk
3
Prioritize
Risk register · Risk appetite vs tolerance · Heat maps · Residual risk after controls
4
Treat (4 T's)
Transfer (insurance/3rd party) · Avoid (stop activity) · Mitigate (implement controls) · Accept (acknowledge residual risk)
5
Monitor & Review
Continuous monitoring, KRI dashboards, reassess after control changes, audit findings, repeat
▸ RISK MATRIX — LIKELIHOOD × IMPACT
LOW Impact MED Impact HIGH Impact CRITICAL Impact
HIGH Likelihood MEDIUM HIGH CRITICAL CRITICAL
MED Likelihood LOW MEDIUM HIGH CRITICAL
LOW Likelihood LOW LOW MEDIUM HIGH

🔧 CompTIA Troubleshooting Methodology 6-STEP CYCLE ▶
Universal Methodology
6-Step CompTIA Troubleshooting Process
Tested across ALL CompTIA certifications. Applies to hardware, software, networking, and security issues. Can loop back to Step 3 if theory fails to resolve the issue.
STEP 1
Identify the Problem
Gather info · Question users · Review documentation · Check environmental changes · Reproduce the problem
▼
STEP 2
Establish a Theory of Probable Cause
Question the obvious first · Consider multiple causes · Research if needed · Use OSI model top-down or bottom-up
▼
STEP 3
Test the Theory
Confirm → proceed to Step 4 · Not confirmed → reestablish theory (back to Step 2) or escalate
▼
STEP 4
Establish a Plan of Action & Implement
Check corporate policies · Reference vendor docs · Backup before changes · Schedule downtime if needed
▼
STEP 5
Verify Full System Functionality
Test the fix · Confirm with user · Implement preventive measures · Confirm no new issues
▼
STEP 6
Document Findings, Actions & Outcomes
Create knowledge base entry · Inform stakeholders · Close ticket · Update runbook/playbook

🗂️ Linux Filesystem Hierarchy (FHS) DIRECTORY TREE ▶
Diagram — Directory Tree
Filesystem Hierarchy Standard — Linux Directory Structure
Everything begins at root /. The FHS defines the purpose of each directory. Knowing where config files, logs, and binaries live is essential for Linux+ and Linux administration.
/ Root of the entire hierarchy
├─ bin/ Essential user binaries: ls, cp, mv, grep, cat
├─ sbin/ System admin binaries: fdisk, iptables, init, fsck
├─ etc/ System-wide config files: passwd, fstab, hosts, ssh/
├─ home/ User home directories: /home/alice, /home/bob
├─ root/ Root user's home — NOT inside /home
├─ var/ Variable data — logs, spool, mail, databases
└─ var/log/ System logs: auth.log, syslog, kern.log
├─ tmp/ Temporary files; typically cleared on reboot
├─ usr/ Secondary hierarchy — non-essential user programs
├─ usr/bin/ Non-essential user binaries: git, python3, vim
├─ usr/lib/ Shared libraries for /usr/bin and /usr/sbin
└─ usr/share/ Arch-independent data: man pages, icons
├─ lib/ Shared libraries for /bin & /sbin; kernel modules
├─ dev/ Device files: /dev/sda, /dev/null, /dev/random
├─ proc/ Virtual FS: /proc/cpuinfo, /proc/meminfo, /proc/PID/
├─ sys/ Virtual FS (sysfs): hardware & kernel device info
├─ mnt/ Temporary mount point for filesystems
├─ media/ Auto-mounted removable media: USB, CD/DVD
├─ opt/ Optional / third-party software packages
├─ srv/ Service data: web root, FTP content
└─ boot/ Bootloader files: vmlinuz, initrd.img, grub/

🚀 Linux Boot Process (BIOS/UEFI → systemd) BOOT CHAIN ▶
Sequence Diagram
Full Linux Boot Chain — Firmware to Login
1 — BIOS / UEFI Firmware
Hardware POST · Locate bootable device via boot order · UEFI reads EFI System Partition (ESP)
2 — MBR / GPT Partition Table
Read first sector for MBR (legacy) or GUID Partition Table (UEFI) · Locate boot partition
3 — GRUB2 Bootloader
Presents boot menu · Loads compressed kernel image (vmlinuz) + initramfs into RAM
4 — Kernel Initialization
Decompresses itself · Probes hardware · Loads drivers · Mounts initramfs as temporary root
5 — initramfs / initrd
Temporary root FS in RAM · Provides drivers needed to mount real root filesystem from disk
6 — systemd (PID 1)
First real process · Reads unit files · Manages targets (runlevels) · Starts services in parallel
7 — Login / TTY / Display Manager
getty for console · GDM/LightDM for GUI · User session starts · PAM handles authentication

🔤 Regular Expressions (Regex) PATTERN MATCHING ▶
Reference
Regex Syntax — Anchors · Classes · Quantifiers · Groups
Regex is used across all platforms — grep/sed/awk on Linux, PowerShell, JavaScript, Python, SQL. Uses PCRE (Perl Compatible) or POSIX syntax depending on engine.
▸ ANCHORS & ASSERTIONS
Token Meaning Example
^ Start of string/line ^Hello matches "Hello world" but not "Say Hello"
$ End of string/line world$ matches "Hello world" but not "world class"
\b Word boundary \bcat\b matches "cat" not "catch"
\B Non-word boundary \Bcat\B matches "scatter" not "cat"
▸ CHARACTER CLASSES
Token Meaning Equivalent
. Any character except newline c.t → "cat","cot","cut"
\d Digit [0-9] \d{3} → "123","456"
\D Non-digit [^0-9]
\w Word char [a-zA-Z0-9_] \w+ → "hello_world"
\W Non-word character [^a-zA-Z0-9_]
\s Whitespace (space, tab, newline) [ \t\n\r\f\v]
\S Non-whitespace [^ \t\n\r\f\v]
[abc] Character class (a, b, or c) [aeiou] matches vowels
[^abc] Negated class (not a, b, or c) [^0-9] matches non-digits
[a-z] Range (lowercase a to z) [A-Za-z] matches any letter
▸ QUANTIFIERS
Token Meaning Greedy vs Lazy
* 0 or more a* greedy; a*? lazy
+ 1 or more a+ greedy; a+? lazy
? 0 or 1 (optional) colou?r → "color" or "colour"
{n} Exactly n times \d{4} → 4-digit number
{n,} n or more times \w{3,} → 3+ word chars
{n,m} Between n and m times \d{2,4} → 2–4 digits
▸ GROUPS & ALTERNATION
Token Meaning Example
(abc) Capture group (\d{3})-(\d{4}) captures area code & number
(?:abc) Non-capture group (?:https?|ftp):// (group without capture)
(?<name>...) Named capture group (?<year>\d{4}) — reference with \k<year>
| Alternation (OR) cat|dog → "cat" or "dog"
(?=...) Positive lookahead \d+(?= dollars) matches "100" in "100 dollars"
(?!...) Negative lookahead \d+(?! dollars) skips dollar amounts
(?<=...) Positive lookbehind (?<=\$)\d+ matches digits after $
▸ FLAGS / MODIFIERS
Flag Meaning Usage
i Case-insensitive /hello/i matches "Hello","HELLO"
g Global (find all matches) /\d+/g finds all numbers
m Multiline (^ $ per line) /^start/m matches each line start
s Dotall (. matches newline) /a.b/s matches "a\nb"
x Verbose (allow comments) Extended mode for readable patterns
▸ COMMON PATTERNS
# Email address ^[\w.+\-]+@[\w\-]+\.[\w.\-]+$ # IPv4 address ^((?:25[0-5]|2[0-4]\d|[01]?\d\d?)\.){3}(?:25[0-5]|2[0-4]\d|[01]?\d\d?)$ # URL (simplified) https?://[\w\-.]+\.[a-z]{2,}(?:/[\w\-._~:/?#[\]@!$&'()*+,;=]*)? # Strong password: 8+ chars, upper, lower, digit, special ^(?=.*[A-Z])(?=.*[a-z])(?=.*\d)(?=.*[\W_]).{8,}$ # MAC address (colon-separated) ^([0-9A-Fa-f]{2}:){5}[0-9A-Fa-f]{2}$

Reference
SQL — DDL · DML · DQL · Joins · Aggregation · Security
SQL is divided into: DDL (Data Definition — CREATE/ALTER/DROP) · DML (Data Manipulation — INSERT/UPDATE/DELETE) · DQL (Data Query — SELECT) · DCL (Data Control — GRANT/REVOKE) · TCL (Transaction Control — COMMIT/ROLLBACK).
DDL — DATA DEFINITION
-- Create table with constraints CREATE TABLE users ( id INTEGER PRIMARY KEY AUTOINCREMENT, username VARCHAR(50) NOT NULL UNIQUE, email VARCHAR(255) NOT NULL UNIQUE, password CHAR(60) NOT NULL, -- bcrypt hash role VARCHAR(20) DEFAULT 'viewer', created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP, active BOOLEAN DEFAULT TRUE ); -- Indexes for performance CREATE INDEX idx_users_email ON users (email); CREATE UNIQUE INDEX idx_users_name ON users (username); -- Alter table ALTER TABLE users ADD COLUMN last_login TIMESTAMP; ALTER TABLE users DROP COLUMN legacy_field; -- Foreign key reference CREATE TABLE posts ( id INTEGER PRIMARY KEY, user_id INTEGER REFERENCES users(id) ON DELETE CASCADE, title TEXT NOT NULL );
DML — INSERT / UPDATE / DELETE
-- Insert INSERT INTO users (username, email, password, role) VALUES ('alice', 'alice@example.com', '$2b$12$...', 'admin'); -- Insert multiple rows INSERT INTO users (username, email, password) VALUES ('bob', 'bob@example.com', 'hash1'), ('eve', 'eve@example.com', 'hash2'); -- Update with WHERE (always include WHERE!) UPDATE users SET role = 'admin', last_login = CURRENT_TIMESTAMP WHERE id = 42; -- Delete (dangerous without WHERE) DELETE FROM users WHERE active = FALSE AND created_at < '2024-01-01';
DQL — SELECT & JOINS
-- Basic SELECT with filtering & ordering SELECT username, email, role, created_at FROM users WHERE active = TRUE AND role IN ('admin', 'editor') AND created_at >= '2025-01-01' ORDER BY created_at DESC LIMIT 25 OFFSET 0; -- JOIN types SELECT u.username, p.title, p.created_at FROM users u INNER JOIN posts p ON u.id = p.user_id -- matching rows only LEFT JOIN comments c ON p.id = c.post_id -- all posts, null if no comment WHERE u.active = TRUE; -- Aggregation SELECT role, COUNT(*) AS user_count, MAX(created_at) AS latest_signup FROM users WHERE active = TRUE GROUP BY role HAVING COUNT(*) > 5 -- filter after GROUP BY ORDER BY user_count DESC; -- Subquery SELECT * FROM users WHERE id IN (SELECT user_id FROM posts WHERE views > 1000); -- Window function SELECT username, role, RANK() OVER (PARTITION BY role ORDER BY created_at) AS rank_in_role FROM users;
