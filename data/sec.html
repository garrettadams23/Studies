🎟️ Kerberos Authentication Flow 4-STEP PROTOCOL ▶
Protocol — Source: CEH + CISSP Cheat Sheets
Kerberos — Ticket-Based Authentication (port 88)
Kerberos uses symmetric + asymmetric encryption. Default in Active Directory environments. Attacks: Kerberoasting (offline crack service ticket), Pass-the-Ticket (reuse TGT/TGS), Golden Ticket (forged TGT using KRBTGT hash).
KDC
Key Distribution Centre — central trust authority (runs AS + TGS)
AS
Authentication Service — validates identity; issues TGT
TGS
Ticket Granting Service — issues service tickets upon valid TGT
TGT
Ticket Granting Ticket — proof of identity; valid for session
1
Client → KDC (AS)
Client sends AS-REQ with username in cleartext. No password sent yet.
2
KDC (AS) → Client
AS responds with TGT encrypted with KRBTGT hash. Client decrypts AS-REP using their password-derived key. If incorrect password → cannot decrypt → fail.
3
Client → KDC (TGS)
Client presents TGT to TGS, requests service ticket for specific service (TGS-REQ).
4
KDC (TGS) → Client → Service
TGS issues service ticket encrypted with service account's hash. Client presents ticket to target service. Access granted without re-entering password.
🏢 Active Directory — Structure, Objects & Attacks SEC+ · CySA+ · PenTest+ ▶
Microsoft Directory Service
AD DS — Authentication, Authorisation & Policy at Enterprise Scale
Active Directory Domain Services (AD DS) is Microsoft's centralised identity and access management system. Nearly every enterprise Windows environment runs it. AD is the #1 target in corporate intrusions — owning AD means owning the network.
▸ STRUCTURE — LOGICAL HIERARCHY
ObjectDescriptionAnalogy
Forest Top-level security boundary. Contains one or more domains. Shares schema and global catalog. The country
Domain Core admin unit. Has its own policies, database (NTDS.dit), and DNS namespace (corp.local). The state
Organisational Unit (OU) Container inside a domain. Groups objects for delegation and GPO application. The department
Domain Controller (DC) Server running AD DS. Authenticates users, stores NTDS.dit, replicates with other DCs. The HR office
Global Catalog (GC) Partial replica of ALL objects in the forest. Speeds up cross-domain lookups. Port 3268/3269. The central index
NTDS.dit The AD database file on each DC. Contains all objects including password hashes. Primary target. The master HR file
Trust Allows users in one domain/forest to access resources in another. Can be one- or two-way, transitive or non-transitive. Inter-company agreement

GPOs are collections of settings pushed to computers/users in an OU. Applied order: Local → Site → Domain → OU (LSDOU). Later policies override earlier ones. Enforced GPOs cannot be overridden. Block Inheritance stops parent GPOs filtering down.

🔒 TLS — Handshake, Cipher Suites & Certificate Chain SEC+ · SECX ▶
Transport Layer Security
How HTTPS Actually Works — Step by Step
TLS (formerly SSL) provides confidentiality (encryption), integrity (MAC), and authentication (certificates) over any transport. Used by HTTPS (443), SMTPS (465), IMAPS (993), and most modern protocols. Current standard: TLS 1.3 (2018).
▸ TLS 1.2 HANDSHAKE (4 ROUND TRIPS)
1
ClientHello
Client → Server. Sends: TLS version, random number (client_random), list of supported cipher suites, compression methods, extensions (SNI, ALPN, supported curves).
2
ServerHello + Certificate
Server → Client. Selects cipher suite and TLS version. Sends server_random. Sends its X.509 certificate chain. Optionally sends ServerKeyExchange (for DHE/ECDHE).
3
Client Key Exchange
Client validates the certificate chain. Generates pre-master secret. If RSA: encrypts with server's public key. If DHE/ECDHE: sends client DH public value. Both sides derive the master secret and session keys independently.
4
Finished (both sides)
Both send a Finished message: HMAC of the entire handshake transcript, encrypted with the new session keys. If both verify correctly — encrypted data transfer begins.
▸ TLS 1.3 IMPROVEMENTS (2 ROUND TRIPS → 1)
FeatureTLS 1.2TLS 1.3
Handshake RTTs2 RTT (full) / 1 RTT (resumption)1 RTT (full) / 0 RTT (resumption — risky)
Key ExchangeRSA or DHE/ECDHEECDHE only — RSA key exchange removed
Forward SecrecyOptional (only with DHE/ECDHE)Mandatory — every session gets unique keys

▸ KEY CONCEPTS
ConceptMeaning
Forward Secrecy (PFS)Each session uses ephemeral keys. Compromising the private key LATER does not decrypt past sessions. Requires DHE or ECDHE.
HSTSHTTP Strict Transport Security. Browser header telling clients to always use HTTPS for N seconds. Defeats SSL stripping.
Certificate PinningApp hardcodes the expected cert/public key. Defeats MITM even with a trusted rogue CA. Used in mobile apps.
OCSP StaplingServer attaches (staples) a pre-fetched OCSP response to handshake. Faster than client fetching OCSP separately.
SNIServer Name Indication. Client sends hostname in ClientHello so server knows which cert to present. Allows virtual hosting.
CT (Cert Transparency)All public certs must be logged in public append-only logs. Lets anyone detect mis-issuance.
