🔤 Regular Expressions (Regex) PATTERN MATCHING ▶
Reference
Regex Syntax — Anchors · Classes · Quantifiers · Groups
Regex is used across all platforms — grep/sed/awk on Linux, PowerShell, JavaScript, Python, SQL. Uses PCRE (Perl Compatible) or POSIX syntax depending on engine.
▸ ANCHORS & ASSERTIONS
Token Meaning Example
^ Start of string/line ^Hello matches "Hello world" but not "Say Hello"
$ End of string/line world$ matches "Hello world" but not "world class"
\b Word boundary \bcat\b matches "cat" not "catch"
\B Non-word boundary \Bcat\B matches "scatter" not "cat"
▸ CHARACTER CLASSES
Token Meaning Equivalent
. Any character except newline c.t → "cat","cot","cut"
\d Digit [0-9] \d{3} → "123","456"
\D Non-digit [^0-9]
\w Word char [a-zA-Z0-9_] \w+ → "hello_world"
\W Non-word character [^a-zA-Z0-9_]
\s Whitespace (space, tab, newline) [ \t\n\r]
\S Non-whitespace [^\t\n\r]
[abc] Character class (a, b, or c) [aeiou] matches vowels
[^abc] Negated class (not a, b, or c) [^0-9] matches non-digits
[a-z] Range (lowercase a to z) [A-Za-z] matches any letter
▸ QUANTIFIERS
Token Meaning Greedy vs Lazy
* 0 or more a* greedy; a*? lazy
+ 1 or more a+ greedy; a+? lazy
? 0 or 1 (optional) colou?r → "color" or "colour"
{n} Exactly n times \d{4} → 4-digit number
{n,} n or more times \w{3,} → 3+ word chars
{n,m} Between n and m times \d{2,4} → 2–4 digits
▸ GROUPS & ALTERNATION
Token Meaning Example
(abc) Capture group (\d{3})-(\d{4}) captures area code & number
(?:abc) Non-capture group (?:https?|ftp):// (group without capture)
(?<name>...) Named capture group (?<year>\d{4}) — reference with \k<year>
| Alternation (OR) cat|dog → "cat" or "dog"
(?=...) Positive lookahead \d+(?= dollars) matches "100" in "100 dollars"
(?!...) Negative lookahead \d+(?! dollars) skips dollar amounts
(?<=...) Positive lookbehind (?<=\$)\d+ matches digits after $
▸ FLAGS / MODIFIERS
Flag Meaning Usage
i Case-insensitive /hello/i matches "Hello","HELLO"
g Global (find all matches) /\d+/g finds all numbers
m Multiline (^ $ per line) /^start/m matches each line start
s Dotall (. matches newline) /a.b/s matches "a b"
x Verbose (allow comments) Extended mode for readable patterns
▸ COMMON PATTERNS
# Email address ^[\w.+\-]+@[\w\-]+\.[\w.\-]+$ # IPv4 address ^((?:25[0-5]|2[0-4]\d|[01]?\d\d?)\.){3}(?:25[0-5]|2[0-4]\d|[01]?\d\d?)$ # Strong password: 8+ chars, upper, lower, digit, special ^(?=.*[A-Z])(?=.*[a-z])(?=.*\d)(?=.*[\W_]).{8,}$ # MAC address (colon-separated) ^([0-9A-Fa-f]{2}:){5}[0-9A-Fa-f]{2}$

🔗 URL Encoding — Percent Encoding Reference ENCODING REFERENCE ▶
Reference — W3Schools · MDN · Google · Sigma
URL / Percent Encoding — Rules, Characters & Interactive Decoder
URLs can only contain a safe subset of ASCII characters. Any character outside that set must be percent-encoded: replaced with %XX where XX is the two-digit uppercase hex value of the UTF-8 byte. Also called percent-encoding (RFC 3986).
▸ ENCODING RULES
Safe Characters
Never encoded: A–Z a–z 0–9 - _ . ~
Also safe in specific URL parts: ! * ' ( )
Reserved Characters
Have special URL meaning — encode when used as data:
: / ? # [ ] @ ! $ & ' ( ) * + , ; =
Always Encode
Spaces, Unicode, control chars, and anything outside ASCII printable range must always be percent-encoded. Space → %20 (or + in query strings only)
Query String Rule
In application/x-www-form-urlencoded (HTML forms), space becomes +. In all other contexts use %20.

🧱 Object-Oriented Programming — Classes, Objects & the 4 Pillars OOP ▶
Paradigm
OOP — Modelling the World as Objects
OOP organises code around objects — bundles of related data (fields / properties) and behaviour (methods / functions). A class is the blueprint; an object (instance) is a copy built from that blueprint. Used in Python, Java, C#, C++, JavaScript, Go, Ruby, and most modern languages.
▸ CLASS vs OBJECT — THE KEY DISTINCTION
# CLASS = blueprint. Defined once. Lives in your source file. # OBJECT = instance. Created at runtime. Can have thousands. # Blueprint (Class) class Dog: # __init__ is the CONSTRUCTOR — runs when object is created def __init__(self, name, breed): self.name = name # PROPERTY (instance data) self.breed = breed def speak(self): # METHOD (behaviour) return f"{self.name} says Woof!" # Objects (Instances) dog1 = Dog("Rex", "Husky") # new Dog object in memory dog2 = Dog("Bella","Poodle") # different object, same class print(dog1.speak()) # → Rex says Woof! print(dog2.name) # → Bella (dot notation accesses properties) # dog1 and dog2 share the CLASS but have INDEPENDENT DATA. # Changing dog1.name does NOT affect dog2.name.
▸ THE FOUR PILLARS OF OOP
1. Encapsulation
Bundle data + methods together. Hide internal state — only expose a controlled interface (public vs private). Why: protects data from accidental corruption. The object is a black box; you call methods, not raw variables.
2. Inheritance
A child class inherits properties & methods from a parent class. Child can extend or override them. Why: reuse code without copy-pasting. Cat extends Animal → Cat automatically has all Animal behaviour.
3. Polymorphism
"Many forms." The same method name behaves differently depending on the object it's called on. animal.speak() returns "Woof" for Dog, "Meow" for Cat. Why: write one piece of code that works with any subtype.
4. Abstraction
Hide complex implementation; expose only what the user needs. You call car.start() without knowing about fuel injectors. Why: reduces cognitive load and prevents misuse of internals.

🔀 Git — Version Control Reference DEVOPS · DAILY USE ▶
Distributed VCS
Git — Track, Branch, Merge, and Collaborate
Git is the universal version control system. It tracks every change to every file in a project, lets multiple people work simultaneously via branches, and merges everything back together. A repository (repo) is the folder Git watches. A commit is a saved snapshot.
▸ THE THREE AREAS OF GIT
Working Directory
Your actual files on disk. Changes here are untracked until you stage them.
Staging Area (Index)
git add moves changes here. A holding zone — you choose exactly what goes into the next commit.
Repository (.git)
git commit permanently saves the staged snapshot here. The full history lives in this hidden folder.
▸ DAILY WORKFLOW COMMANDS
# ── Setup ────────────────────────────────────────────────── git config --global user.name "Your Name" git config --global user.email "you@example.com" git init # create new local repo git clone <url> # copy remote repo locally # ── Core workflow ────────────────────────────────────────── git status # what's changed? what's staged? git add file.txt # stage one file git add . # stage ALL changed files git commit -m "feat: add login" # save snapshot with message git push origin main # upload commits to remote git pull # fetch + merge remote changes # ── Branching ────────────────────────────────────────────── git branch # list branches git switch -c feature/login # create + switch to new branch git switch main # go back to main git merge feature/login # merge branch into current git branch -d feature/login # delete merged branch # ── History & inspection ─────────────────────────────────── git log --oneline --graph # visual commit history git diff # unstaged changes git diff --staged # staged vs last commit git show <hash> # what changed in that commit # ── Undo ─────────────────────────────────────────────────── git restore file.txt # discard unstaged changes (safe) git restore --staged file.txt # unstage a file (safe) git revert <hash> # undo commit, keep history (safe) git reset --hard <hash> # ⚠ destroy history to this point git stash # temporarily shelve WIP changes git stash pop # restore shelved changes
