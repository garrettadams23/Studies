🗂️ Linux Filesystem Hierarchy (FHS) DIRECTORY TREE ▶
Diagram — Directory Tree
Filesystem Hierarchy Standard — Linux Directory Structure
Everything begins at root /. The FHS defines the purpose of each directory.
/ Root of the entire hierarchy
├─ bin/ Essential user binaries: ls, cp, mv, grep, cat
├─ sbin/ System admin binaries: fdisk, iptables, init, fsck
├─ etc/ System-wide config files: passwd, fstab, hosts, ssh/
├─ home/ User home directories: /home/alice, /home/bob
├─ root/ Root user's home — NOT inside /home
├─ var/ Variable data — logs, spool, mail, databases
└─ var/log/ System logs: auth.log, syslog, kern.log
├─ tmp/ Temporary files; typically cleared on reboot
├─ usr/ Secondary hierarchy — non-essential user programs
├─ usr/bin/ Non-essential user binaries: git, python3, vim
├─ usr/lib/ Shared libraries for /usr/bin and /usr/sbin
└─ usr/share/ Arch-independent data: man pages, icons
├─ lib/ Shared libraries for /bin & /sbin; kernel modules
├─ dev/ Device files: /dev/sda, /dev/null, /dev/random
├─ proc/ Virtual FS: /proc/cpuinfo, /proc/meminfo, /proc/PID/
├─ sys/ Virtual FS (sysfs): hardware & kernel device info
├─ mnt/ Temporary mount point for filesystems
├─ media/ Auto-mounted removable media: USB, CD/DVD
├─ opt/ Optional / third-party software packages
├─ srv/ Service data: web root, FTP content
└─ boot/ Bootloader files: vmlinuz, initrd.img, grub/

🚀 Linux Boot Process (BIOS/UEFI → systemd) BOOT CHAIN ▶
Sequence Diagram
Full Linux Boot Chain — Firmware to Login
1 — BIOS / UEFI Firmware
Hardware POST · Locate bootable device via boot order · UEFI reads EFI System Partition (ESP)
2 — MBR / GPT Partition Table
Read first sector for MBR (legacy) or GUID Partition Table (UEFI) · Locate boot partition
3 — GRUB2 Bootloader
Presents boot menu · Loads compressed kernel image (vmlinuz) + initramfs into RAM
4 — Kernel Initialization
Decompresses itself · Probes hardware · Loads drivers · Mounts initramfs as temporary root
5 — initramfs / initrd
Temporary root FS in RAM · Provides drivers needed to mount real root filesystem from disk
6 — systemd (PID 1)
First real process · Reads unit files · Manages targets (runlevels) · Starts services in parallel
7 — Login / TTY / Display Manager
getty for console · GDM/LightDM for GUI · User session starts · PAM handles authentication
🔐 SSH — Commands, Keys & Tunneling COMPLETE REFERENCE ▶
Reference — Source: SSH Cheat Sheet
SSH · SCP · SFTP · ssh-keygen · Tunneling
Secure Shell listens on TCP port 22 by default. Uses asymmetric crypto for key exchange, symmetric for session data. Replaces Telnet/FTP/rsh. Config files: ~/.ssh/config (client) · /etc/ssh/sshd_config (server).
BASIC CONNECTION COMMANDS
# Connect to remote host ssh user@hostname # Default port 22 ssh -p 3344 user@hostname # Custom port ssh -i ~/.ssh/key.pem user@1.2.3.4 # Specific identity file ssh user@host 'ls -la /var/log' # Execute remote command inline ssh user@host bash < script.sh # Run local script remotely # Compress traffic (useful on slow links) ssh -C user@hostname # X11 forwarding (run GUI apps remotely) ssh -X user@hostname # Trusted X11 forwarding ssh -Y user@hostname # Less secure — full display forward
SSH KEY MANAGEMENT (ssh-keygen)
# Generate key pair (RSA 4096-bit, labeled with email) ssh-keygen -t rsa -b 4096 -C "you@email.com" # Generate modern Ed25519 key (preferred — smaller, faster) ssh-keygen -t ed25519 -C "you@email.com" # Other key management commands ssh-keygen -F hostname # Check if host is in known_hosts ssh-keygen -R hostname # Remove host from known_hosts ssh-keygen -y -f private.key # Derive public key from private key ssh-keygen -p -f ~/.ssh/id_rsa # Change key passphrase ssh-copy-id user@server # Install public key to remote host # Key files reference # ~/.ssh/id_rsa — private key (protect with chmod 600) # ~/.ssh/id_rsa.pub — public key (safe to share) # ~/.ssh/known_hosts — trusted hosts (auto-populated) # ~/.ssh/authorized_keys — keys permitted to log in here
SCP — SECURE COPY
scp user@server:/remote/file.txt ./local/ # Remote → local scp ./local/file.txt user@server:/remote/ # Local → remote scp -r user@server:/folder ./local/ # Recursive (directory) scp -P 8022 user@server:/file ./ # Custom port (-P uppercase) scp -C user@server:/big_file ./ # Enable compression scp user1@host1:/file user2@host2:/dest # Server-to-server
SSH TUNNELING — PORT FORWARDING
# LOCAL (-L): Forward local_port → remote_host:remote_port via SSH server # Access internal service on remote network as if it were local ssh -L 8080:internal-db:5432 user@jump-host # → localhost:8080 now reaches internal-db port 5432 # REMOTE (-R): Forward remote_port on SSH server → local machine # Expose a local service through a remote public server ssh -R 9090:localhost:3000 -N -f user@public-server # → public-server:9090 now reaches your local port 3000 # DYNAMIC (-D): Create a SOCKS proxy on local_port # Route browser/app traffic through SSH server ssh -D 8888 -q -C -N -f user@remote-host # → Set browser SOCKS5 proxy to 127.0.0.1:8888 # PROXY JUMP (-J): Multi-hop through bastion host ssh -J user@bastion user@internal-host ssh -J user@hop1:port1,user@hop2:port2 user@target # Multi-hop # Flags used in tunneling # -N No remote commands — dedicated tunnel only # -f Fork to background after authentication # -q Quiet mode — suppress all warnings # -C Compress tunnel data

Sessions · Windows · Panes · Copy Mode
Tmux (Terminal Multiplexer) runs persistent sessions that survive SSH disconnects. Install: sudo apt install tmux (Debian/Ubuntu) · brew install tmux (macOS). Prefix key: Ctrl+b (default) — press before every shortcut.
Session
A persistent collection of pseudo-terminals. Survives disconnect. Multiple windows inside.
Window
A single screen view (like a tab). Can contain multiple panes.
Pane
A split section of a window. Each runs its own terminal independently.
SESSION COMMANDS (from CLI)
tmux # Start new session tmux new -s mysession # New named session tmux ls # List all sessions tmux a # Attach to last session tmux a -t mysession # Attach to named session tmux kill-session -t s0 # Kill named session tmux kill-session -a # Kill ALL except current

💾 RAID Levels Reference STORAGE REDUNDANCY ▶
Reference — Source: A+ Cheat Sheet
Redundant Array of Independent Disks — Levels 0, 1, 5, 6, 10
RAID provides fault tolerance and/or performance. Does NOT replace backups. Minimum disk counts and failure tolerance differ by level.
Level Name Min Disks Can Lose Characteristic Use Case
RAID 0 Striping 2 0 Data split across disks; combined read/write speed. No redundancy. Scratch disks, video editing (speed > safety)
RAID 1 Mirroring 2 1 Exact copy on each disk. Half usable capacity. Fast reads, slower writes. OS drives, critical single-point protection
RAID 5 Striping + Parity 3 1 Data + parity distributed. (N-1) usable capacity. Rebuild from parity on failure. File servers, NAS; good balance of speed/redundancy
RAID 6 Double Parity 4 2 Like RAID 5 but two parity blocks. (N-2) usable capacity. Slower writes. Large arrays where double failure is a real risk
RAID 10 Stripe of Mirrors 4 1 per mirror Mirrored pairs then striped. 50% usable capacity. Excellent performance + redundancy. Databases, high-performance high-availability workloads
⚠️ RAID is NOT backup. RAID protects against drive failure only — not against ransomware, accidental deletion, corruption, or site disasters. Follow the 3-2-1 rule: 3 copies · 2 different media types · 1 offsite.

▸ LASER PRINTER IMAGING PROCESS (7 STEPS)
1 — Processing
Page data received; printer RIP (Raster Image Processor) converts to bitmap image in memory
2 — Charging
Primary corona wire/charge roller applies uniform negative charge (~-600V) to photosensitive drum
3 — Exposing
Laser beam neutralizes charge on drum where text/images will appear, creating a latent electrostatic image
4 — Developing
Negatively charged toner is attracted to the uncharged (exposed) areas of the drum; visible image forms
5 — Transferring
Transfer corona belt/roller applies positive charge to paper, attracting toner from drum to paper
6 — Fusing
Fuser assembly (heating element + pressure roller) melts toner permanently into paper fibers (~180°C)
7 — Cleaning
Cleaning blade scrapes residual toner off drum; erase lamp neutralizes drum charge for next cycle

Package Once, Run Anywhere — Container Concepts
Docker packages an application and ALL its dependencies into a container — a lightweight, isolated process that runs identically everywhere. An image is the read-only blueprint; a container is a running instance of an image.

Use caseFull OS isolation, legacy appsMicroservices, CI/CD, cloud-native

⚙️ Kubernetes — Container Orchestration Reference DEVOPS · CLOUD ▶
K8s — Google Borg → Open Source
Deploy, Scale, and Heal Containers Automatically
Kubernetes (K8s) automates deployment, scaling, and management of containerised applications. It clusters multiple machines into one compute pool and decides where each container runs. Control Plane = the brain. Worker Nodes = where workloads run.
